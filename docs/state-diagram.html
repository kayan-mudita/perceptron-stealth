Create Video Record
POST /api/generate
FAST path. Creates DB record, plans cut composition (pure logic, no API calls). Returns immediately.
Inputs
  • videoId?: string (optional, reuse existing)
  • model: string (default "kling_2.6")
  • script: string
  • format: string (e.g. "talking_head_15")
  • photoId?: string
  • voiceId?: string
Outputs
  • video: { id, status: "generating", ... }
  • composition: { format, totalCuts, totalDuration, status: "queued" }
Internal Logic
  • enforceUsageLimit() — plan-based video cap
  • generateLimiter.check(10, userId) — rate limit
  • planComposition(format, script) — pure logic
  • Prisma Video.create() or Video.update()
  • Cut patterns defined in video-compositor.ts
click
S
Status Polling
GET /api/generate/status?videoId=...
Returns pipeline progress (step, currentCut, percent). Auto-marks as failed after 10min timeout.
Inputs
  • videoId: string (query param)
Outputs
  • video: { id, status, videoUrl, thumbnailUrl, model, title, duration }
  • progress: { step, currentCut, totalCuts, percent }
  • error?: string (when status = "failed")
Progress Calculation
  • queued: 0% | expand: 5% | tts: 15% | anchor: 20%
  • cut: 20-85% (proportional to cut index)
  • stitch: 90% | store: 95% | done: 100%
  • Timeout: 10 min → auto-fail
Pipeline Steps — POST /api/generate/process — One step per call (avoids 26s Netlify timeout)
1 step=expand — Prompt Expansion via Gemini
POST /api/generate/process { videoId, step: "expand" }
Expands every cut's template prompt into a production-grade video prompt using Gemini 2.5 Flash. Each cut gets character context, anti-glitch rules, and model-specific instructions.
Inputs
  • videoId: reads script, model, contentType from DB
  • planComposition(format, script) → cut templates
  • Character context: sheets, brand profile, user name
  • Model-specific instructions (Kling, MiniMax, Wan, LTX)
External Services
  • Gemini 2.5 Flash — expandPrompt() per cut (parallel)
  • System prompt: 150-200 word production prompt with iPhone camera specs, lived-in environment, character description, anti-glitch rules
  • Returns JSON: { expandedPrompt, script, title, estimatedDuration }
Outputs (stored in sourceReview JSON)
  • cuts[]: { index, type, duration, generateDuration, prompt }
  • format: string
  • originalScript: string (preserved for TTS)
  • nextStep: "tts"
2 step=tts — Voice Generation (TTS)
POST /api/generate/process { videoId, step: "tts" }
Generates voiceover audio from originalScript. Uses the best available TTS provider. Non-fatal: pipeline continues without audio if all providers fail.
Inputs
  • sourceReview.originalScript (not the expanded prompts)
  • Minimum 10 characters to trigger TTS
External Services (Priority Chain)
  • MiniMax via FAL — fal-ai/minimax/speech-02-hd (best quality)
  • MiniMax Standalone API — api.minimax.chat/v1/t2a_v2 (fallback)
  • ElevenLabs API — eleven_multilingual_v2 (fallback)
  • Supabase downloadAndStore() for TTS audio persistence
Outputs
  • sourceReview.ttsAudioUrl: string | null
  • hasAudio: boolean
  • nextStep: "cut", nextCutIndex: 0
3 step=cut&cutIndex=i — Generate Video Cut via FAL
POST /api/generate/process { videoId, step: "cut", cutIndex: i }
LOOP Repeats for each cut (0..N-1). Typical: 3-4 cuts for 15s, 8 cuts for 30s.
Submits one video generation job to FAL. Resolves anchor image (starting frame → user photo fallback). Returns immediately with job ID.
Inputs
  • cut.prompt: expanded production prompt
  • cut.generateDuration: seconds (always > final duration)
  • photoUrl: starting frame || user's primary photo
  • model: FAL model ID (e.g. "kling_2.6")
External Services
  • FAL POST queue.fal.run/{model-id}
  • Models: Kling 2.6 Pro, MiniMax Video 01, Hailuo 2.3 Fast, Wan 2.1, LTX 2.3
  • Returns: request_id, status_url, response_url
  • Job ID stored as "FAL::{status_url}::{response_url}"
  • Supabase downloadAndStore() if sync result
Outputs
  • status: "cut_submitted" | "cut_done"
  • jobId: string (FAL composite ID)
  • cutJobs[i]: { jobId, status, videoUrl, thumbnailUrl, trimTo }
  • nextStep: "poll" (if async) | "cut" (next cut) | "stitch"
4 step=poll&cutIndex=i — Poll FAL Job Status
POST /api/generate/process { videoId, step: "poll", cutIndex: i }
POLL Called every 3s by frontend until cut completes. Max 120 attempts (6min per cut).
Checks FAL job status. On completion: downloads video from FAL temp URL and persists to Supabase Storage for permanence.
Inputs
  • cutJobs[i].jobId: "FAL::{status_url}::{response_url}"
External Services
  • FAL GET {status_url} — check IN_QUEUE / IN_PROGRESS / COMPLETED / FAILED
  • FAL GET {response_url} — fetch result on COMPLETED
  • Supabase downloadAndStore() video + thumbnail to permanent URLs
Outputs
  • status: "polling" | "cut_done" | "cut_failed"
  • videoUrl: Supabase permanent URL (on completion)
  • nextStep: "poll" (retry) | "cut" (next cut) | "stitch"
  • retryAfter: 5 (seconds)
5 step=stitch — Submit Stitch to Shotstack
POST /api/generate/process { videoId, step: "stitch" }
Collects all completed cut URLs, builds a Shotstack timeline with trim points and audio overlay, submits render job. Single-cut videos skip Shotstack entirely.
Inputs
  • cutJobs[*].videoUrl: Supabase URLs for all completed cuts
  • cutJobs[*].trimTo: seconds to keep from each cut
  • sourceReview.ttsAudioUrl: voiceover audio (optional)
  • aspectRatio: "9:16" (vertical)
External Services
  • Shotstack POST /edit/{env}/render
  • Timeline: video clips on track 0, audio on track 1
  • Cross-dissolve transitions between cuts
  • Output: MP4, HD (720p), 30fps, high quality
  • If single cut: uses cut directly as final video (no Shotstack)
  • If Shotstack not configured: falls back to first cut
Outputs
  • status: "stitch_submitted" | "done"
  • jobId: Shotstack render ID
  • nextStep: "poll_stitch" (if async)
6 step=poll_stitch — Poll Shotstack Render
POST /api/generate/process { videoId, step: "poll_stitch" }
POLL Called every 5s by frontend. Max 120 attempts (10min).
Checks Shotstack render status. On completion: downloads stitched video, persists to Supabase, marks video as "review" status.
Inputs
  • sourceReview.stitchJobId: Shotstack render ID
External Services
  • Shotstack GET /edit/{env}/render/{jobId}
  • States: queued → processing → rendering → finalizing → completed | failed
  • Supabase downloadAndStore() final video + thumbnail
Outputs
  • status: "polling" | "done" | "failed"
  • videoUrl: Supabase permanent URL (on done)
  • Prisma Video.update({ status: "review", videoUrl, thumbnailUrl })
✓ Video Complete — Ready for Review
Video marked as "review" in DB. Supabase Storage holds permanent URLs for video and thumbnail. Frontend shows cinematic reveal.
Final State
  • video.status: "review"
  • video.videoUrl: Supabase public URL (.mp4)
  • video.thumbnailUrl: Supabase public URL (.jpg)
  • sourceReview: JSON metadata with all cut info
Status Lifecycle
  • generating → review → approved → published
  • generating → failed (on error or 10min timeout)

Key Architecture Decisions
Step-per-Call Pipeline
Each pipeline step is a separate HTTP call to /api/generate/process. This avoids Netlify's 26-second function timeout. The frontend drives the pipeline by calling steps sequentially and polling for async results.
Multi-Cut Composition
Every video is multiple AI generation calls stitched together. Each cut generates MORE than needed (e.g. 8s for a 3s cut) to pick the best segment. "If you splice it together, it'll actually look good."
FAL-First Pipeline
All video generation routes through FAL. One API key, one billing, access to every model (Kling, MiniMax, Wan, LTX, Runway). Adding a new model = one line in FAL_MODELS registry.
Permanent URL Strategy
FAL returns temporary URLs that expire. Every completed asset (video cuts, thumbnails, TTS audio, stitched video) is immediately downloaded and re-uploaded to Supabase Storage for permanent public URLs.
Starting Frame Consistency
A single anchor image (generated via Gemini) is used as the reference for every video cut. Ensures the same person appears consistently across all segments. Generated during onboarding, reused across all future videos.
Graceful Degradation
TTS failure is non-fatal (video continues without audio). Shotstack failure falls back to first cut. Starting frame failure falls back to user photo. 10-minute timeout auto-marks stuck videos as failed.
